# Main 통합 개발 — PRD (요구사항 정리 및 정책 결정 문서)

이 문서는 `../CLAUDE.md`(목표/메뉴 명세), `docs/Plan.md`(phase 계획), 4개 PoC의
`docs/Plan.md`를 종합해, Main 구현에 들어가기 전에 **주의해야 할 부분**,
**아직 결정되지 않은 정책**, **PoC 간 충돌 가능 지점**을 정리한다. 이미
`Plan.md`에서 결정된 사항(예: Repository fail-soft 정책)은 중복 서술하지
않고 링크만 남긴다.

## 1. 이미 결정된 사항 (재확인)

- Repository 손상 데이터 처리: `Json/`의 fail-soft 정책 채택. 단 `Add` 중복 ID는
  fail-hard(`std::invalid_argument`) 유지. → `Plan.md` "PoC 간 확인된 정책 차이" 절.
- 계층 구조: Model(순수 도메인) → Repository(영속성) → Controller(흐름) → View(콘솔 I/O).
  Controller는 추상 View 인터페이스에만 의존(`MVC/` 패턴).
- 재고 판정 경계값(`Monitor/` `StockStatusJudge` 정책):
  `stock == 0` → DEPLETED(최우선), `0 < stock < demand` → SHORTAGE,
  `stock >= demand` → SUFFICIENT(동률 포함, `demand == 0` 포함). 음수는 0 clamp.
- 생산 큐 계산: 부족분 = 주문수량 − 재고, 실생산량 = `ceil(부족분 / 수율)`,
  총생산시간 = 평균생산시간 × 실생산량.
- 시간 소스는 `IClock` 추상화로 주입(도메인 로직에서 `system_clock::now()` 직접 호출 금지).

## 2. 정책 결정 현황

### 2-1~2-3. 재고 차감 시점 / demand 정의 / 승인 판정 기준 — **결정 완료**

- **물리적 `Stock` 증감은 두 시점에서만 일어난다** (원래 도메인 규칙인
  "생산완료 시 재고에 실생산량 반영"은 그대로 유지되며, 아래 두 번째
  항목이 이번에 새로 결정된 부분이다):
  - 생산 완료(`PRODUCING -> CONFIRMED`) 시: `Stock += 실생산량` (물리적
    입고 — 웨이퍼가 실제로 창고에 들어오는 시점).
  - 출고(`CONFIRMED -> RELEASE`) 시: `Stock -= 주문수량` (물리적 출고).
  - 승인 시점(재고 충분 → 즉시 `CONFIRMED`)에는 `Stock`을 변경하지 않는다.
- **승인(재고 충분 → `CONFIRMED`) 시점에는 해당 수량을 내부적으로 "예약(선점)"
  처리한다.** 다른 주문이 이미 선점된 재고를 다시 "충분"으로 오판해 중복
  확보하지 못하게 막는 것이 목적. 예약 여부는 별도 필드가 아니라 "미출고
  `CONFIRMED` 주문들의 수량 합"으로 파생 계산한다.
- 따라서 승인 시 재고 충분/부족 판정은 물리적 `Stock` 값이 아니라
  **가용재고 = `Stock` − (아직 출고되지 않은 `CONFIRMED` 주문들의 수량 합)**
  을 기준으로 한다.
- 모니터링의 `demand = RESERVED + CONFIRMED + PRODUCING` 수량 합은 그대로
  유효하다.
- **구현 함의**: `Sample`에 물리적 `Stock` 외에 "가용재고 계산" 로직이 필요.
  이는 `Sample` 엔티티 자체의 필드가 아니라, 승인 시점에 해당 `SampleId`의
  미출고 `CONFIRMED` 주문 수량 합을 Repository/Controller에서 조회해 계산하는
  파생값으로 설계한다(Model의 순수 함수로 `가용재고 = Stock - 미출고예약수량`
  형태만 두고, 예약수량 집계는 Controller가 Order 목록에서 계산해 넘겨준다).

#### 2-3-1. (파생 이슈) 생산 완료로 생긴 실생산량 중 "초과분"의 처리 — **결정 완료**
- 실생산량은 `ceil(부족분 / 수율)`로, 수율이 1 미만이면 부족분보다 많이
  생산된다(예: 부족분 10, 수율 0.9 → 실생산량 12, 초과분 2).
- **결정**: 초과분은 생산 완료 즉시 가용재고에 반영되어야 한다. 위 공식
  (`가용재고 = Stock − 미출고 CONFIRMED 수량 합`)을 그대로 적용하면 별도
  특례 로직 없이 자동으로 성립한다 — 생산완료 시 `Stock`은 실생산량만큼
  늘고, 예약 합계는 그 주문의 원 수량만큼만 늘어나므로, 차이(초과분)가
  가용재고 증가분으로 자연히 반영된다.

### 2-4. 생산 완료 처리 트리거 방식 — **결정 완료**
- **결정**: 시간 경과 기반 자동 완료. `IClock` 기준으로 큐 선두 항목의
  누적 총생산시간이 경과하면 자동으로 `PRODUCING -> CONFIRMED` 전환된다.
- 구현 함의: 메뉴 진입(생산라인 조회 등) 시점마다 "현재 시각 vs 예정 완료
  시각"을 비교해 완료 여부를 판정하는 로직이 필요하다(폴링형 체크). 정확한
  체크 시점(예: 생산라인 메뉴 진입 시, 또는 메인 메뉴 진입 시마다)은 Phase 3
  설계에서 구체화한다.

### 2-5. 시료 등록 시 재고(Stock) 초기값 — **결정 완료**
- **결정**: 등록 직후 재고는 항상 0. 이후 생산 완료를 통해서만 재고(및
  가용재고)가 증가한다.

### 2-8. 생산 큐 아이템 단위 — **결정 완료**
- **결정**: 주문(Order) 1:1 단위. PRODUCING 상태 주문 하나당 큐 아이템
  하나이며, 여러 주문을 하나의 생산 배치로 병합하지 않는다.

### 2-6. 수율(Yield)의 의미와 입력 방식
- 도메인 정의상 "수율 = 정상 시료 수 / 총 생산 시료 수"로, 생산 실적에서
  계산되는 값처럼 보이지만, 메뉴 명세는 시료 등록 시점에 사용자가 직접 값을
  입력하도록 되어 있다(추정 수율). 즉 이 시스템은 수율을 실측 갱신하지 않고
  등록 시 고정값으로 취급한다는 뜻인데, 이것이 의도된 단순화인지 확인 필요.
- **결정**: 등록 시 입력받은 값을 고정값으로만 취급한다. 이후 생산 실적으로
  갱신하는 기능은 두지 않는다(미생산 낭비/실측 수율 반영 기능은 범위 밖).

### 2-7. ID 채번 전략의 적용 범위 — **결정 완료**
- `SampleId`는 메뉴 명세상 사용자가 직접 입력(`S-001`형). `OrderId`는
  형식(`ORD-YYYYMMDD-0043`)이 정해져 있으나 사용자가 입력하는 필드가 아니므로
  시스템이 자동 채번해야 한다.
- **결정**: 일자별 순번 리셋. `YYYYMMDD`는 주문 생성일(`IClock` 기준)로
  채우고, 순번은 같은 날짜 안에서 기존 최댓값 + 1(자정이 지나 날짜가
  바뀌면 `0001`부터 재시작). `Dummy/`의 `IdAllocator`(기존 최댓값+1, 파싱
  실패 항목 무시) 전략을 날짜 스코프로 재구현한다.

### 2-9. 비정상 종료(강제 종료) 후 재시작 시 데이터 무결성 — **결정 완료**
- **요구사항**: 프로그램이 정상적으로 `0. 종료`를 거치지 않고 중간에
  강제 종료(콘솔 강제 닫기, 프로세스 kill 등)되더라도, 그 시점까지 반영된
  모든 데이터는 JSON 파일에 저장되어 있어야 하고, 재시작 시 그 데이터를
  문제없이 로드해 이어서 사용할 수 있어야 한다.
- **결정**: 아래 두 가지를 만족해야 "재시작 안전"이 성립한다고 정의한다.
  1. **지연 저장 금지**: Sample/Order를 변경하는 모든 Repository 동작
     (`Add`/`Update`/`Delete`)은 메모리만 바꾸고 끝나는 것이 아니라, 호출
     즉시 파일에 저장(`Save()`)한다. 메뉴 동작 하나가 끝나면 그 결과는
     항상 디스크에 반영되어 있어야 하며, "나중에 한꺼번에 저장" 같은
     지연/배치 저장은 두지 않는다.
  2. **일시적 상태(메모리 전용)도 재시작 후 복원 가능해야 한다**: 생산
     큐(FIFO)처럼 프로세스 메모리에만 있는 상태라도, 그 상태가 사라지면
     정상 흐름이 막히는 경우(예: `PRODUCING` 주문이 영원히 완료되지 않음)는
     허용하지 않는다. 그런 상태를 재구성하는 데 필요한 최소한의 값(예:
     생산 시작 시각, 부족분/실생산량/총생산시간)은 해당 `Order`에 함께
     저장해 두고, 앱 시작 시 Repository의 데이터만으로 그 상태를 다시
     계산/복원한다.
- **구현 함의**: `Order`에 `ProductionShortage`/`ProductionActualQuantity`/
  `ProductionTotalTimeMinutes`/`ProductionStartedAt`(모두 `PRODUCING`일 때만
  값이 있는 `std::optional`)를 추가하고, `ProductionLineController`가
  생성될 때 Repository의 `PRODUCING` 주문들을 시작 시각(FIFO) 순으로 큐에
  복원한다. 이 결정은 `docs/phase5-integration.md`의 시나리오 7 수동 검증
  중 실제로 재현된 버그(재시작 시 생산 큐 소실)를 고치면서 확정되었다.
- **범위 밖**: 파일 쓰기 도중 프로세스가 강제 종료되어 JSON 파일 자체가
  중간까지만 쓰인 채로 남는 극단적 경우(원자적 쓰기/임시 파일-rename)까지
  방어하는 것은 이 결정의 범위가 아니다. `Json/`의 fail-soft 정책(손상된
  JSON은 예외 없이 빈 값/원소 skip으로 처리)이 이런 경우에도 프로그램이
  죽지 않게 해주지만, 그 시점까지의 데이터 일부가 유실될 가능성 자체를
  없애지는 않는다.

## 3. 충돌 가능 지점 (PoC 간 설계 차이로 인해 재구현 시 혼선 우려)

| 항목 | PoC별 차이 | Main 재구현 시 주의점 |
|---|---|---|
| 손상 데이터 처리 | `Json`(fail-soft) vs `Monitor`/`Dummy`(fail-hard) | 이미 결정됨(1절). 코드 작성 시 `Monitor`/`Dummy` 스타일의 예외 던지는 습관이 섞여 들어가지 않도록 주의. |
| View 인터페이스 분리 단위 | `MVC/`는 시료 전용 `ISampleView` 1개만 검증. Main은 6개 메뉴 전체를 다뤄야 함 | 메뉴별로 인터페이스를 잘게 쪼갤지, 하나로 통합할지 Phase 3에서 결정(과도한 추상화 금지 원칙과 충돌 가능). |
| 재고 판정 `demand` 구성 | `Monitor/`는 자체 PoC 내 임의 스키마로 집계 로직만 검증(Order 상태 구성비를 Main과 동일하게 가정하지 않았을 수 있음) | Main의 실제 5개 상태 기준으로 `demand` 정의를 다시 검증해야 한다(2-2 참고). |
| ID/시간 포맷 | `Json`은 `FormatIso8601`/`ParseIso8601`을 자체 구현, `Dummy`도 별도 `JsonIo.h`에서 유사 기능 구현 | 두 PoC가 동일한 포맷(ISO 8601, 초 단위)을 썼는지 재확인 후 Main에서 하나로 통일. |
| 재고 차감 책임 소재 | 4개 PoC 어디에도 "차감 시점" 정책이 없음(전부 이 개념을 다루지 않음) | 2-1이 실질적으로 신규 정책이며 PoC 참고 없이 Main이 처음 결정해야 하는 사안. |

## 4. 다음 단계

  생산완료 자동 판정(경과시간 비교) 순수 함수, `OrderId` 일자별 채번 로직.
- Phase 2: `Order`에 예약/미출고 여부를 조회하기 위한 Repository 질의(예:
  `FindBySampleIdAndStatus` 또는 Controller에서 `GetAll()` 후 필터링) 필요

- Phase 3: 승인 로직이 가용재고 기준으로 판정하도록 수정, 생산라인 메뉴

## 5. 확정된 정책 요약 (Decision Log)

| 항목 | 결정 |
|---|---|
| 물리적 `Stock` 증감 시점 | 증가는 생산완료 시(`+= 실생산량`, 변경 없음), 감소는 출고 시(`-= 주문수량`, 신규 결정) |
| 승인(`CONFIRMED`) 시점 재고 처리 | 물리적 차감 없음, 내부적으로 "예약(선점)" 처리 |

| 생산 큐 아이템 단위 | 주문(Order) 1:1 |
| 수율(Yield) 갱신 여부 | 등록 시 고정값, 이후 갱신 없음 |
