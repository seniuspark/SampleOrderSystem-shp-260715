# 반도체 시료 생산주문관리 시스템 — 최종 개발 (Main)

이 디렉토리는 4개 PoC(`MVC`, `Json`, `Monitor`, `Dummy`)에서 검증된 설계를
통합하여 **완성된 반도체 시료 생산주문관리 콘솔 시스템**을 개발하는 곳이다.
독립된 Git Repository로 제출한다 (`SampleOrderSystem-영문이름-사번`).
요구사항 원문은 `../docs/[CRA_AI] Day3_개인과제_반도체시료관리.pdf`,
공통 도메인 모델은 상위 `../CLAUDE.md`를 참고.

## 이 프로젝트가 통합하는 4가지 축

| 축 | 참고 PoC | 통합 내용 |
|---|---|---|
| 구조 | `MVC/` | Model/Controller/View 계층 분리 |
| 영속성 | `Json/` | 시료/주문 JSON 저장·로딩, CRUD |
| 모니터링 | `Monitor/` | 상태별 주문 집계, 재고 여유/부족/고갈 판정 |
| 테스트 데이터 | `Dummy/` | 개발/시연용 더미 데이터 생성 |

각 PoC의 코드를 그대로 복사해 붙이지 말고, 검증된 설계와 인터페이스를 참고하여
이 프로젝트의 요구사항에 맞게 다시 구현한다 (TDD로).

## 배경 (S-Semi)

가상의 반도체 회사 S-Semi는 다양한 반도체 시료(Sample)를 생산해 연구소,
팹리스 업체, 대학 연구실에 납품한다. 시료는 주문이 들어오면 웨이퍼 공정
설비를 통해 제작되고, 검수 후 고객에게 출고된다. 엑셀/메모장 기반 관리의
한계(주문 처리 여부 파악 불가, 공정 완료 시점 불명확, 불필요한 추가 생산)를
해결하기 위해 이 시스템을 개발한다.

역할별 흐름: 고객 → (이메일로 시료 요청) → 주문 담당자 → (주문서 작성/전달) →
생산 담당자 → (승인/거절, 개발 시료 등록).

## 메인 메뉴 및 기능 명세

메인 메뉴는 전체 시료 요약 정보(등록 시료 수, 총 재고, 전체 주문 수, 생산라인
대기 등)를 함께 보여준다.

### 1. 시료 관리
| 기능 | 설명 |
|---|---|
| 시료 등록 | 시료 ID, 이름, 평균 생산시간, 수율을 입력받아 등록 |
| 시료 조회 | 등록된 모든 시료 목록 + 현재 재고 수량 표시 |
| 시료 검색 | 이름 등 속성으로 특정 시료 검색 |

수율 = 정상 시료 수 / 총 생산 시료 수 (예: 100개 중 90개 정상 → 0.9).

### 2. 시료 주문 (예약, RESERVED 생성)
- 입력값: 시료 ID, 고객명, 주문 수량
- 등록된 시료(존재하는 SampleId)에 대해서만 주문 가능
- 생성 즉시 상태는 `RESERVED`

### 3. 주문 승인/거절
- `RESERVED` 상태 주문 목록을 표시
- **승인**: 재고 확인 후 자동 분기
  - 재고 충분 → 즉시 `CONFIRMED`
  - 재고 부족 → 생산 라인(FIFO 큐)에 자동 등록, 상태 `PRODUCING`
- **거절**: 즉시 `REJECTED` (정상 흐름 외, 모니터링에서 제외)

### 4. 모니터링
- **주문량 확인**: 상태별(RESERVED/CONFIRMED/PRODUCING/RELEASE) 건수. REJECTED 제외
- **재고량 확인**: 시료별 현재 재고 + 상태(여유/부족/고갈)

### 5. 생산라인
- 생산 라인은 단일 라인, 시료를 하나씩 생산, FIFO 스케줄링
- 부족분 = 주문 수량 - 현재 재고
- 실 생산량 = `ceil(부족분 / 수율)`
- 총 생산 시간 = `평균 생산시간 * 실 생산량`
- 생산 완료 시 재고에 실 생산량 반영, 해당 주문 `PRODUCING -> CONFIRMED`
- 화면에 현재 생산 중 항목(주문 정보, 진행 상황 등 자율 결정)과 FIFO 대기열을 표시

### 6. 출고 처리
- `CONFIRMED` 상태 주문만 대상
- 특정 주문 선택 → 출고 실행 → 상태 `RELEASE`

## 주문 상태 흐름 요약

```
RESERVED --거절--> REJECTED
RESERVED --승인(재고충분)--> CONFIRMED --출고--> RELEASE
RESERVED --승인(재고부족)--> PRODUCING --생산완료--> CONFIRMED --출고--> RELEASE
```

## 기술 스택

- C++20, Visual Studio(.vcxproj), x64/x86, Debug/Release
- GoogleTest는 NuGet으로 설치, TDD로 개발
- 콘솔 UI, JSON 파일 기반 영속성 (앱 재시작 후에도 데이터 유지)
- 시간 관련 계산(생산 완료 예정 등)은 테스트 가능하도록 시간 소스를 주입/추상화
  한다 (실제 `std::chrono::system_clock` 직접 호출을 도메인 로직에 흩뿌리지 않는다).

## 평가/작업 주안점 (미션2 기준)

1. **문서 관리**: 이 `CLAUDE.md` 외에 필요 시 `PRD.md`, ADR 등을 이 디렉토리에
   유지하고 최신 상태로 관리한다.
2. **Harness 도입**: gtest 실행 환경(테스트 프로젝트 구성, CI까지는 아니어도 로컬
   실행 가능한 형태)을 갖춘다.
3. **Test**: 기능 명세의 각 분기(승인 시 재고충분/부족, 생산완료, 출고 등)를
   빠짐없이 테스트로 커버한다.
4. **Clean Code**: 계층 분리 유지, 불필요한 추상화·주석 금지.
5. **Commit 이력**: 기능 단위로 작게 커밋하여 개발 과정이 드러나도록 한다.

## 개발 순서 제안

1. Model 계층: Sample/Order 엔티티, 상태 enum, 생산 큐 계산 로직(순수 함수)부터
   TDD로 시작한다.
2. Repository(영속성) 계층: `Json/` PoC에서 검증한 설계를 재구현.
3. Controller 계층: 각 메뉴 흐름(시료 등록/주문/승인·거절/모니터링/생산라인/출고)을
   가짜 View로 테스트하며 구현.
4. View 계층: 실제 콘솔 입출력 연결. 예시 UI(PDF)는 참고용이며 화면 구성은 자유.
5. `Dummy/`에서 검증한 생성 로직으로 초기 데모 데이터를 채워 수동 시나리오 검증.
